Qu'est-ce qu'Annuaire Congo ?

La République du Congo compte des milliers d'entreprises locales dont la visibilité numérique reste très limitée, faute d'un annuaire professionnel centralisé et fiable. Annuaire Congo répond à ce manque en proposant une plateforme SaaS complète qui permet à trois types d'acteurs d'interagir :

  • Les propriétaires d'entreprise créent et enrichissent leur fiche (secteur, localisation, contacts, services, documents, images), souscrivent à un abonnement premium et suivent leurs statistiques d'audience.
  • Les utilisateurs recherchent des entreprises par secteur, région ou mot-clé, consultent les fiches enrichies, et interagissent avec CongoBot — l'assistant IA intégré — pour affiner leur recherche.
  • Les administrateurs valident les fiches soumises, modèrent les signalements, gèrent les plans d'abonnement et supervisent la plateforme via un journal d'audit complet.

La plateforme s'articule autour de six piliers fonctionnels, tous exposés via l'API REST et consommés par le frontend Angular :

search

Recherche avancée

Filtres combinés (secteur, région, statut, premium) avec pagination côté serveur et tri dynamique. La recherche textuelle est enrichie par CongoBot.

Cartographie interactive

Visualisation Leaflet géolocalisée des entreprises sur les 12 régions du Congo, avec clustering de marqueurs et fiche pop-up.

Score de confiance 0–100 calculé sur 10 indicateurs (complétude, vérification, engagement…), affiché directement sur chaque fiche publique.

smart_toy

CongoBot

Assistant conversationnel propulsé par LLaMA 3.3 70B (Groq API), accessible sans authentification pour guider les visiteurs vers les bonnes entreprises.

analytics

Analytics propriétaire

Tableau de bord dédié aux propriétaires : vues de profil, clics sur les contacts, évolution mensuelle et export CSV des données d'audience.

payments

Abonnements & paiements

Plans Free / Premium / Business avec intégration MTN MoMo, Airtel Money et Stripe. Machine à états pour le cycle de vie des abonnements.

engineering
Points forts de l'implémentation

Le backend applique la Clean Architecture stricte (Domain → Application → Infrastructure → Api) sans dépendances circulaires. Chaque commande ou requête transite par un pipeline MediatR de 4 behaviors chaînés : gestion des exceptions → validation FluentValidation → monitoring des performances → cache HybridCache (.NET 9, L1 in-process 30 s + L2 distribué 10 min). L'observabilité est assurée par Serilog → Seq (logs structurés) et OpenTelemetry → Prometheus → Grafana (métriques).

Rôles et niveaux d'accès

La plateforme distingue quatre rôles aux périmètres bien délimités. Les routes API sont sécurisées par des attributs [Authorize(Roles = "…")] ; certains endpoints publics portent explicitement [AllowAnonymous] (recherche, fiches, CongoBot, Trust Score).

RegularUser Utilisateur authentifié Profil personnel, notifications, favoris, consultation de l'annuaire et des fiches enrichies

Démarrage / Démarrage rapide

Démarrage rapide

Suivez ces étapes pour lancer la plateforme en mode développement sur votre machine locale.

1. Cloner le dépôt

bash
git clone https://github.com/your-org/ANNUAIRECONGO.git
cd ANNUAIRECONGO

2. Lancer le backend (.NET 10)

bash
# Restaurer les dépendances
dotnet restore

# Lancer les migrations et le seed
cd src/ANNUAIRECONGO.Api
dotnet run
lightbulb
Astuce

L'API lance automatiquement les migrations et le seed de la base de données au démarrage (via ApplicationDbContextInitialiser). L'API écoute sur https://localhost:7139 et http://localhost:5000.

3. Lancer le frontend (Angular 19)

bash
cd src/ANNUAIRECONGO.Client
npm install
ng serve

L'application Angular sera disponible sur http://localhost:4200.

4. Comptes de test pré-configurés

Rôle Email Mot de passe
Admin admin@localhost admin@localhost
EntrepriseOwner john.businessOwner@localhost Password123!

5. Lancer avec Docker Compose

bash
# Démarrage complet : API + SQL Server + Seq + Prometheus + Grafana
docker compose up -d

# Vérifier les services
docker compose ps
Service URL Description
API .NET http://localhost:5001 Backend REST API
Angular App http://localhost:80 Frontend (via Nginx)
SQL Server localhost:1433 Base de données
Seq http://localhost:8081 Centralisation des logs
Prometheus http://localhost:9090 Métriques
Grafana http://localhost:3000 Tableaux de bord visuels

Flux de données principal

1
Client Angular

L'utilisateur interagit avec l'interface. Les requêtes HTTP sont envoyées via ApiService avec le token JWT en en-tête.

2
API Controller

Le contrôleur reçoit la requête, effectue les vérifications d'autorisation, et envoie une commande/requête via MediatR.

3
Handler CQRS

Le handler traite la logique applicative. La pipeline MediatR exécute les validators FluentValidation avant le handler.

4
Domaine

Les entités du domaine appliquent les règles métier. Les transitions d'état sont validées (ex : Draft → Pending → Active).

5
Infrastructure

EF Core persiste les changements. Les domain events sont dispatched après SaveChangesAsync.

Architecture / Clean Architecture

Clean Architecture — Détails

Chaque couche a des responsabilités bien définies. Les dépendances pointent uniquement vers l'intérieur.

Couche Domain

Contient les entités métier, les value objects, les domain events, et les règles de validation. Aucune dépendance externe.

Namespace Entités principales Responsabilité
Companies Company, CompanyContact, CompanyImage, CompanyDocument, CompanyService, CompanyReport Gestion des fiches entreprises avec machine d'état (Draft → Pending → Active/Rejected/Suspended)
BusinessOwners BusinessOwner Profil du propriétaire d'entreprise avec vérification KYC
Subscriptions Subscription, Payment, Plan Abonnements avec cycle de vie (Pending → Active → ExpiringSoon → Expired/Cancelled)
Geography Region, City Découpage géographique du Congo-Brazzaville (12 régions, 35+ villes)
Sectors Sector, CompanySector Classification sectorielle (6 secteurs stratégiques)
Analytics ProfileView, ContactClick, AnalyticsDailySummary Suivi des interactions utilisateurs
Notifications Notification Notifications in-app avec types (Info, Success, Warning, Error)
Logs AdminLog Journal d'audit des actions administratives
Identity Role, RefreshToken Rôles et tokens de rafraîchissement

Couche Application

Contient les handlers CQRS (via MediatR), les validators (FluentValidation), les behaviors (pipeline MediatR), et les interfaces des services.

info
Pattern CQRS

Chaque cas d'utilisation est implémenté via une commande (écriture) ou une requête (lecture) MediatR. Les validators FluentValidation s'exécutent automatiquement via le ValidationBehavior de la pipeline MediatR.

Couche Infrastructure

Implémente les interfaces définies dans Application. Comprend :

storage

Persistance (EF Core)

AppDbContext avec 20+ DbSets, intercepteurs d'audit, configurations Fluent API, et dispatch automatique des domain events.

fingerprint

Identity & JWT

IdentityService et TokenProvider gèrent l'inscription, la connexion, les tokens JWT avec refresh, et la vérification email.

smart_toy

Service IA (Grok)

GrokService intègre LLaMA 3.3 70B pour la recherche sémantique, le Trust Score, et les rapports IA.

folder_open

Stockage local

LocalStorageService gère l'upload de fichiers (logos, images, documents) avec validation de taille (max 10 MB).

notifications

Notifications

NotificationService crée des notifications in-app pour les événements métier (validation, paiement, expiration).

receipt_long

Facturation

InvoiceService génère les factures PDF pour les paiements confirmés.

Couche API

Point d'entrée HTTP. Comprend 7 contrôleurs, le middleware pipeline, le rate limiting (100 req/min), et l'output caching (avec exclusion des requêtes authentifiées).

Couche Contracts

DTOs partagés entre l'API et l'Application. Définit les structures de requête/réponse utilisées par les contrôleurs et les handlers.

Architecture / Modèle de domaine

Modèle de domaine

Entités, agrégats et machines d'état du domaine métier.

Machine d'état — Company

L'entité Company est l'agrégat racine principal. Son statut suit une machine d'état stricte :

Draft
→ Submit()
Pending
↗ Approve() ↘ Reject(reason)
Active
Rejected
↓ Suspend()
Suspended
warning
Transitions protégées

Chaque méthode de transition valide l'état actuel et retourne un Result<Updated>. Les transitions invalides retournent une erreur typée (CompanyErrors).

Machine d'état — Subscription

P
Pending

Créée lors de la souscription. En attente de confirmation de paiement.

A
Active

Activée après confirmation du paiement. L'entreprise bénéficie des fonctionnalités du plan.

E
ExpiringSoon

Notification envoyée quand l'expiration approche.

X
Expired / Cancelled

L'abonnement est terminé. Renouvellement possible.

Architecture / Pipeline CQRS

Pipeline CQRS & Behaviors

Chaque commande et requête MediatR traverse un pipeline de 4 behaviors avant d'atteindre le handler. Cette architecture garantit la séparation des préoccupations et évite la duplication de code transversal.

info
Enregistrement automatique

Les behaviors sont enregistrés dans Application/DependencyInjection.cs via AddMediatR(). Les validators FluentValidation sont découverts automatiquement par AddValidatorsFromAssembly().

Ordre d'exécution du pipeline

1
ExceptionHandlingBehavior<TRequest, TResponse>

Enveloppe toute la pipeline dans un try/catch. Convertit les exceptions non gérées en Result.Failure avec message d'erreur structuré. Évite que les exceptions remontent jusqu'au contrôleur.

2
ValidationBehavior<TRequest, TResponse>

Cherche tous les IValidator<TRequest> enregistrés et les exécute. Si des règles échouent, retourne immédiatement une 400 Bad Request avec la liste complète des erreurs. Le handler n'est jamais appelé si la validation échoue.

3
PerformanceBehavior<TRequest, TResponse>

Mesure le temps d'exécution total du handler avec Stopwatch. Si la durée dépasse 500 ms, émet un Warning Serilog avec le nom de la commande, l'ID utilisateur et le temps écoulé. Aide à détecter les régressions de performance.

4
CachingBehavior<TRequest, TResponse>

S'applique uniquement aux requêtes implémentant ICacheableQuery. Vérifie le HybridCache (L1 local 30 s + L2 distribué 10 min) avant d'appeler le handler. Sur un hit de cache, retourne directement le résultat sans toucher à la base de données.

✓
Handler (IRequestHandler<TRequest, TResponse>)

La logique métier du cas d'utilisation. Accède à IAppDbContext, aux services, et retourne un Result<T>. Les domain events sont dispatchés après SaveChangesAsync().

Exemple — Commande complète

C# — CreateCompanyCommand.cs
// Application/Features/Companies/Commands/CreateCompany/
public sealed record CreateCompanyCommand(
    string Name,
    string? Description,
    Guid CityId,
    List<Guid> SectorIds
) : IRequest<Result<Guid>>;
C# — CreateCompanyCommandValidator.cs
public sealed class CreateCompanyCommandValidator
    : AbstractValidator<CreateCompanyCommand>
{
    public CreateCompanyCommandValidator()
    {
        RuleFor(x => x.Name)
            .NotEmpty()
            .MaximumLength(100);

        RuleFor(x => x.CityId)
            .NotEmpty();

        RuleFor(x => x.SectorIds)
            .NotEmpty()
            .WithMessage("Au moins un secteur requis.");
    }
}
C# — CreateCompanyCommandHandler.cs
public sealed class CreateCompanyCommandHandler(
    IAppDbContext ctx,
    IUser user)
    : IRequestHandler<CreateCompanyCommand, Result<Guid>>
{
    public async Task<Result<Guid>> Handle(
        CreateCompanyCommand request, CancellationToken ct)
    {
        var company = Company.Create(
            request.Name,
            request.Description,
            request.CityId,
            user.BusinessOwnerId);   // from JWT claim

        ctx.Companies.Add(company);
        await ctx.SaveChangesAsync(ct);

        return Result.Success(company.Id);
    }
}

Exemple de test unitaire

C# — Domain.UnitTests
[Fact]
public void Submit_WhenDraft_ShouldTransitionToPending()
{
    // Arrange
    var company = Company.Create("Test SARL", null, cityId, ownerId);

    // Act
    var result = company.Submit();

    // Assert
    result.IsSuccess.Should().BeTrue();
    company.Status.Should().Be(CompanyStatus.Pending);
    company.SubmittedAt.Should().NotBeNull();
}

[Fact]
public void Submit_WhenAlreadyPending_ShouldFail()
{
    var company = Company.Create("Test", null, cityId, ownerId);
    company.Submit(); // first submit OK

    var result = company.Submit(); // second attempt

    result.IsFailure.Should().BeTrue();
    result.Error.Code.Should().Be("Company.InvalidState");
}

Guide pour ajouter une nouvelle fonctionnalité end-to-end, des entités à l'endpoint HTTP.

Checklist — Nouvelle fonctionnalité

1
Entité de domaine

Créer Domain/{Groupe}/{Entité}.cs avec setters privés, constructeur EF Core privé et factory method statique.

2
Erreurs de domaine

Créer Domain/{Groupe}/{Entité}Errors.cs avec des constantes Error typées.

3
Configuration EF Core

Créer Infrastructure/Data/Configurations/{Entité}Configurations.cs implémentant IEntityTypeConfiguration<T>.

4
DbSet + Migration

Ajouter DbSet<T> à IAppDbContext et AppDbContext, puis dotnet ef migrations add.

5
Commande/Requête + Validator + Handler

Créer dans Application/Features/{Groupe}/Commands/{Action}/ ou Queries/. Le validator est découvert automatiquement.

6
Contrats (DTOs)

Créer les DTOs request/response dans Contracts/Requests/ et Contracts/Responses/.

7
Endpoint HTTP

Ajouter l'action dans le contrôleur existant ou créer un nouveau contrôleur avec [ApiVersion("1")].

8
Tests

Tester l'entité domaine (UnitTests), le handler (Application.UnitTests avec mocks) et l'endpoint (Api.IntegrationTests).

Conventions de nommage

Type Convention Exemple
Commande {Action}{Entité}Command CreateCompanyCommand
Handler de commande {Action}{Entité}CommandHandler CreateCompanyCommandHandler
Validator {Action}{Entité}CommandValidator CreateCompanyCommandValidator
Requête Get{Entité}{Filtre}Query GetCompanyByIdQuery
Erreurs domaine {Entité}Errors CompanyErrors
Configuration EF {Entité}Configurations CompanyConfigurations

Règle clé : Primary Constructors

C# — Pattern préféré
// ✅ Utiliser les primary constructors C# 13
public sealed class CreateCompanyCommandHandler(
    IAppDbContext ctx,
    IUser user,
    IHybridCache cache)
    : IRequestHandler<CreateCompanyCommand, Result<Guid>>
{ ... }

// ❌ Éviter les champs avec this._
public sealed class CreateCompanyCommandHandler
    : IRequestHandler<CreateCompanyCommand, Result<Guid>>
{
    private readonly IAppDbContext _ctx;
    public CreateCompanyCommandHandler(IAppDbContext ctx) { _ctx = ctx; }
}
